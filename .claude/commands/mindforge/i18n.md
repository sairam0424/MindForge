<objective>
Set up a comprehensive internationalization architecture that externalizes
all user-facing strings, implements ICU message format for complex plurals
and selections, configures locale detection with fallback chains, and
establishes a scalable translation workflow with lazy-loading per route.
</objective>

<execution_context>
@.mindforge/skills/i18n-architecture/SKILL.md
</execution_context>

<context>
$ARGUMENTS
</context>

<process>
1. Audit all user-facing strings in the application: UI labels, error messages, email templates, push notifications, PDF reports, and dynamic content. Flag hardcoded strings, string concatenation (breaks in inflected languages), and embedded formatting (dates, numbers, currencies).
2. Set up message catalog structure organized by feature namespace: one file per locale per namespace (e.g., messages/en/auth.json, messages/fr/auth.json). Keep namespaces small (<200 keys) for efficient lazy-loading. Define key naming convention (dot-separated, descriptive).
3. Implement ICU message format based on --format flag (default ICU): configure plural rules ({count, plural, one {# item} other {# items}}), select for gender/context ({gender, select, male {He} female {She} other {They}}), and nested formatting. Validate messages at build time.
4. Configure locale detection with fallback chain: (1) URL parameter/path prefix, (2) user preference from profile/cookie, (3) Accept-Language header, (4) geo-IP lookup, (5) default locale. Implement locale negotiation (fr-CA falls back to fr, then to en).
5. Add lazy-loading per route based on --lazy-load flag: split message bundles by route/feature, load locale data on navigation (prefetch next likely locale), cache loaded bundles in memory, and handle loading states (show skeleton or previous locale until loaded).
6. Handle RTL layout for Arabic/Hebrew based on --locales: use CSS logical properties (margin-inline-start instead of margin-left), configure dir="rtl" attribute, mirror icons and layouts, and test bidirectional text mixing within a single paragraph.
7. Set up number, date, and currency formatting using Intl API: configure Intl.NumberFormat per locale (decimal separators, grouping), Intl.DateTimeFormat (12h/24h, month names), and Intl.RelativeTimeFormat. Never hardcode format patterns.
8. Configure translation workflow: source strings extracted to translation management system (Crowdin/Phrase/Lokalise), context and screenshots attached per key, translator review and QA process, automated PR with new translations, and CI validation of completeness.
9. Implement locale-aware sorting and searching: configure Intl.Collator for proper alphabetical ordering per locale, handle diacritics in search (normalize to NFD), and support locale-specific search tokenization for CJK languages.
10. Log i18n architecture in AUDIT with: app, locales supported, format standard, lazy-loading strategy, RTL support, translation workflow, and completeness percentage per locale.
</process>
